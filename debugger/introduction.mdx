title: AI Agent Debugger for Long-Running Agents
sidebarTitle: Introduction
---

The Laminar debugger records an agent run, lets you inspect every LLM call and tool use in a transcript, and replays the run from any point with your fix applied. On replay it serves cached LLM responses from the recorded trace up to a boundary you pick, then runs live past it, so you only pay for the calls that actually changed.

## Why it exists

Long-running agents fail slowly. A run takes minutes, the bad call is three-quarters of the way through, and every fix means waiting for the full run again. The feedback loop is too slow to iterate on.

The debugger collapses that loop. Record once, then replay from just before the call you're fixing. The recorded calls before your boundary come back from cache instantly; only the calls after it run live. Every run lands in the same session, so each attempt is a numbered trace you can read, compare, and annotate as you go.

## Where it fits in your development workflow

The debugger is the iterate-on-a-fix loop for your agent, the same way a debugger in your IDE is the loop for ordinary code. The difference is who drives the loop. The debugger is built so the loop is the same whether a coding agent runs it, you run it, or the two of you trade off, and you watch the whole thing from one session view either way.

  <Card title="Coding agent drives" icon="robot">

With the [Laminar skill](https://github.com/lmnr-ai/lmnr-skills) the agent records the run, reads the transcript, finds the bad call, edits your agent, and replays, writing a note before each attempt so you can follow its reasoning in the session view without touching the terminal.
  </Card>
  <Card title="Human in the loop" icon="users">
    Let the agent do the iterating, but stay in the driver's seat. Read each replay as it lands, point the agent at the span that's wrong, adjust the replay boundary yourself, or take over a run by hand. The session is the shared workspace: the agent's notes and your own sit side by side.
  </Card>
  <Card title="You drive" icon="user">
    Skip the coding agent entirely. Every step the agent would take is a CLI command or an environment variable you can run yourself: record with `LMNR_DEBUG=true`, inspect with SQL, replay with two more env vars. The loop and the session view are identical.
  </Card>
</CardGroup>

This matters most when a coding agent is fixing your agent. It can run dozens of replay iterations, each one cheap and fast, without re-running the expensive prefix of the trace every time, while you watch the transcripts land in real time and step in whenever you want to.

## The two primitives

- **Sessions** group every debug run from one investigation. All the traces appear together in the Laminar UI so you (or your coding agent) can compare attempts side by side and follow the notes attached to each run. The first `LMNR_DEBUG=true` run starts a session and writes `.lmnr/debug-session.json`; every later run in the same directory rejoins it automatically, so you never copy a session id between runs.
- **Replay** re-runs the agent against a recorded trace. The debugger serves cached LLM responses up to the boundary you choose and runs everything past it live. See [how caching works](/debugger/caching) for the mechanism.
