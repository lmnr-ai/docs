The Laminar debugger lets you record an agent run, inspect every LLM call and tool use in a transcript, and replay from any point with your fix applied. You only pay for the calls that actually changed.

<Note>
**The recommended way to use the debugger is with an AI coding agent.** Add the [Laminar skill](https://github.com/lmnr-ai/lmnr-skills) to Claude Code, Cursor, or Codex and the agent will run the full record/inspect/replay loop for you. The rest of this page documents the steps for those who want to drive it manually.
</Note>

## Why the debugger

Long-running agents fail slowly. You wait for the full run, find the bad call three-quarters of the way through, fix it, and wait again. Repeat.

The debugger breaks that cycle with two primitives:

- **Sessions**: a named group that collects every debug run you do during an investigation. All traces appear together in the Laminar UI so you can compare attempts side by side.
- **Replay**: on a re-run, the debugger serves cached LLM responses from the source trace up to a boundary you choose. Only the calls past the boundary go live, so you pay only for what changed.

## Prerequisites

Install and instrument the agent first:

- **Instrument your agent**: [TypeScript](/tracing/integrations/overview) or [Python](/tracing/integrations/overview) SDK
- **Install the CLI**: `npm install -g lmnr-cli` or run with `npx lmnr-cli@latest`
- **Make sure your project API key is in your environment** (`LMNR_PROJECT_API_KEY`)

## How your agent uses Debugger

This is what your agent will be instructed to do when using the Laminar skill.

### 1. Record a run

Add `LMNR_DEBUG=true` to your agent's run command:

```bash
LMNR_DEBUG=true node my_agent.js 2>&1 | tee run.log
grep 'LMNR_DEBUG_RUN' run.log
```

```bash
LMNR_DEBUG=true python my_agent.py 2>&1 | tee run.log
grep 'LMNR_DEBUG_RUN' run.log
```

On startup the SDK registers a new debug session with Laminar, logs the session's `debugger_url`, and opens it in your browser. When the run finishes, the SDK prints a `LMNR_DEBUG_RUN` line with the run's ids. Always pipe stdout/stderr and grep for it explicitly, since other output will drown it.

The JSON on that line looks like:

```json
{
  "trace_id": "…",
  "session_id": "…",
  "replay_trace_id": null,
  "cache_until": 0,
  "debugger_url": "https://laminar.run/project/<projectId>/debugger-sessions/<sessionId>",
  "started_at": "…"
}
```

Save the `session_id`. Every subsequent run in this investigation must set `LMNR_DEBUG_SESSION_ID` to it, or each run mints a fresh session that isn't grouped with your investigation and your traces stop appearing together.

### 2. Open the session in Laminar

Click the `debugger_url` from the `LMNR_DEBUG_RUN` line, or open **Debugger** in your project and select the session.

The session view shows every debug run as an expandable trace segment. Each segment renders a transcript: auto-extracted user inputs, LLM turns, tool calls, and collapsed sub-agent cards. Hover an LLM span to copy a ready-to-paste rerun prompt; hover any other span to copy its ID.

### 3. Name the session and annotate each run

The session view shows your `session_id` in the header. Name the investigation once:

```bash
npx lmnr-cli debug session set-name <session-id> "Fix report length + search tool"
```

Write a note **before** each run to explain what you are about to test. Set it via `LMNR_TRACE_METADATA`:

```bash
LMNR_DEBUG=true \
LMNR_DEBUG_SESSION_ID=<session-id> \
LMNR_TRACE_METADATA='{"rollout.note": "## What I am testing\nReplaying calls 1–3, running call 4 live with the new length cap.\n\n"}' \
node my_agent.js 2>&1 | tee run.log
```

The note appears in the session view as markdown above the run's transcript. The human watching the session sees your reasoning in real time as the trace arrives.

After the run completes, append a follow-up:

```bash
npx lmnr-cli trace append-note <trace-id> "## What this run showed
Length cap is working (~180 words, was ~600). Next: check citations."
```

Notes are append-only. Each call adds a paragraph to the existing note.

In notes, reference a specific span with an XML tag. The UI renders it as a clickable chip that opens that span:

```text
<span id='<spanId>' name='synthesis call' />
```

Get the span id from the session view (hover the span row and click **Copy span ID**) or from a SQL query.

### 4. Inspect with SQL

Query the session's traces from the CLI:

```bash
npx lmnr-cli sql query "
  SELECT id AS trace_id, start_time, status, total_tokens
  FROM traces
  WHERE simpleJSONExtractString(metadata, 'rollout.session_id') = '<session-id>'
  ORDER BY start_time DESC
  LIMIT 10"
```

Locate the bad span:

```bash
npx lmnr-cli sql query "
  SELECT span_id, name, span_type, start_time, status
  FROM spans
  WHERE trace_id = '<trace-id>'
  ORDER BY start_time ASC"
```

`span_type` is `LLM`, `TOOL`, `DEFAULT`, or `CACHED` (a replayed call in a replay run). Find the LLM call **before** the buggy one — its `span_id` is the replay boundary you'll pass as `LMNR_DEBUG_CACHE_UNTIL`.

Run `npx lmnr-cli sql schema` to see all available tables.

### 5. Replay to iterate fast

After editing the agent, re-run with replay:

```bash
LMNR_DEBUG=true \
LMNR_DEBUG_SESSION_ID=<session-id> \
LMNR_DEBUG_REPLAY_TRACE_ID=<trace-id> \
LMNR_DEBUG_CACHE_UNTIL=<span-id> \
node my_agent.js 2>&1 | tee run.log
grep 'LMNR_DEBUG_RUN' run.log
```

`LMNR_DEBUG_REPLAY_TRACE_ID` is the `trace_id` from the run you want to replay. `LMNR_DEBUG_CACHE_UNTIL` is the span id of the last LLM call to replay: the debugger serves cached responses through that call (inclusive) and runs live past it. Pass the id of the call **before** the buggy one, and the debugger runs your fix live while caching everything before it. The span id can be a full UUID, the last two UUID groups, the 16-char OTel id, or any hex suffix.

Each replay produces a new trace in the same session, so attempts appear side by side in the UI.

### 6. Replay the last run automatically

`LMNR_DEBUG_FROM_LAST_RUN=true` seeds the config from the previous run's pointer file (`.lmnr/last-run.json` in the working directory). The pointer file is opt-in: it is only written when the run that produced it set `LMNR_DEBUG_WRITE_LAST_RUN_TO_FILE=true`.

```bash
# Record — writes .lmnr/last-run.json when the run finishes
LMNR_DEBUG=true \
LMNR_DEBUG_WRITE_LAST_RUN_TO_FILE=true \
node my_agent.js

# Replay the run above without copying ids
LMNR_DEBUG=true \
LMNR_DEBUG_FROM_LAST_RUN=true \
LMNR_DEBUG_CACHE_UNTIL=<span-id> \
node my_agent.js
```

This replays the most recent trace without having to copy ids. Individual `LMNR_DEBUG_*` vars still override per-field.

### 7. Re-orient with a session summary

After a context reset or to catch up on an ongoing session, dump all trace notes in order:

```bash
npx lmnr-cli debug session summary <session-id>
npx lmnr-cli debug session summary <session-id> --json
```

Output is one block per trace, oldest first: the markdown note followed by a `<trace id="…" end-time="…"/>` tag you can feed into SQL queries.

## Environment variables reference

| Variable | Required | Description |
|---|---|---|
| `LMNR_DEBUG` | Yes | Set to `true`, `1`, `yes`, or `on` to enable debug mode. |
| `LMNR_DEBUG_SESSION_ID` | After first run | Session id from the first run's `LMNR_DEBUG_RUN` line. Required on every run after the first. |
| `LMNR_DEBUG_REPLAY_TRACE_ID` | For replay | Trace id of the run to replay from. |
| `LMNR_DEBUG_CACHE_UNTIL` | For replay | Span id of the last LLM call to replay (inclusive); calls after it run live. Accepts a full UUID, the last two UUID groups, the 16-char OTel id, or any hex suffix. |
| `LMNR_DEBUG_WRITE_LAST_RUN_TO_FILE` | Optional | Write the run pointer to `.lmnr/last-run.json` when the run finishes, so a later run can replay it with `LMNR_DEBUG_FROM_LAST_RUN`. |
| `LMNR_DEBUG_FROM_LAST_RUN` | Optional | Seed replay config from `.lmnr/last-run.json` instead of env vars. Requires the previous run to have set `LMNR_DEBUG_WRITE_LAST_RUN_TO_FILE`. |
| `LMNR_TRACE_METADATA` | Optional | JSON object attached to the trace. Use key `rollout.note` for the pre-run annotation. |
